# QS-010-대시보드-기능-추가-용이성

## 개요

### Quality Scenario ID
QS-010

### 제목
대시보드 기능 추가 용이성

### 설명
대시보드에 새로운 기능(예: 알림 기능, 필터 기능)을 추가할 때 필요한 작업의 범위

### 품질 속성
변경 용이성 (요구사항 변경)

## 환경

### 시스템 상태
- 시스템이 개발 및 유지보수 단계에 있음
- Count 통합 관리 시스템이 Kubernetes 환경에서 MSA로 배포되어 있음
- 기존 대시보드 기능이 구현되어 있음

### 초기 조건
- 대시보드에 새로운 기능 추가 요구사항이 있음
- 예: 알림 기능, 필터 기능, 정렬 기능, 내보내기 기능 등
- 기존 대시보드 기능과 통합되거나 독립적인 새로운 기능
- 관련 컴포넌트 코드가 존재함

### 부하 조건
- 변경 작업이 진행되는 개발 환경
- 변경 영향도를 분석하기 위한 코드베이스 분석이 수행됨

### 관련 컴포넌트
- DashboardUI (Boundary)
- DashboardManager (Control)
- DashboardConfigDB (Entity)
- 새로운 기능 컴포넌트 (추가될 컴포넌트)

## 동작

1. 대시보드에 새로운 기능 추가 요구사항 발생
   - 예: 알림 기능 추가
2. 변경 영향도 분석 수행
   - 새로운 기능의 요구사항 분석
   - 기존 대시보드 구조 파악
   - 새로운 기능 컴포넌트 추가 필요성 확인
   - 기존 컴포넌트 수정 필요성 확인
   - DashboardConfigDB 스키마 변경 필요성 확인
3. 변경 작업 수행
   - 새로운 기능 컴포넌트 개발 (필요한 경우)
   - DashboardManager에 새로운 기능 처리 로직 추가
   - DashboardUI에 새로운 기능 UI 추가
   - DashboardConfigDB 스키마 수정 (필요한 경우)
   - 테스트 코드 작성
   - 문서 업데이트
4. 변경 작업 완료 및 배포

## 측정

### 측정 항목
대시보드 기능 추가 시 수정/영향을 받는 모듈의 크기

### 측정 공식
```
변경 비용 = Σ(LOC(추가되는 모듈)) + Σ(LOC(변경되는 모듈)) + Σ(LOC(변경되는 모듈을 의존하는 모듈))
```

여기서:
- `LOC(추가되는 모듈)`: 새로운 대시보드 기능을 위해 추가되는 모듈의 코드 라인 수(Lines of Code)
- `LOC(변경되는 모듈)`: 대시보드 기능 추가로 인해 직접 수정되는 모듈의 코드 라인 수(Lines of Code)
- `LOC(변경되는 모듈을 의존하는 모듈)`: 변경되는 모듈을 의존하는 모든 모듈의 코드 라인 수(Lines of Code)

### 측정 방법
- 대시보드 기능 추가 시 추가되는 모듈을 식별 (필요한 경우)
- 변경되는 모듈을 식별 (예: DashboardManager, DashboardUI, DashboardConfigDB)
- 변경되는 모듈을 의존하는 모든 모듈을 의존성 그래프를 통해 식별
- 추가되는 모듈의 전체 코드 라인 수를 측정
- 변경되는 모듈의 전체 코드 라인 수를 측정
- 변경되는 모듈을 의존하는 각 모듈의 전체 코드 라인 수를 측정
- 변경 비용은 추가되는 모듈의 크기, 변경되는 모듈의 크기, 의존하는 모듈들의 크기의 합으로 표현
- 측정은 기능의 복잡도, 기존 구조와의 통합 방식에 따라 달라질 수 있음을 고려
